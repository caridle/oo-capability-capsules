# OO 能力胶囊 C01：PRD 生成器

从一句话需求生成 OO 导向的产品需求文档。

## 触发条件

用户提到：PRD、产品需求文档、需求分析、MRD、BRD、软件需求规格、SRS、写需求、需求拆解

## 输入格式

用户提供一句话需求，格式自由。示例：
- "做一个学生选课系统，支持在线选课和成绩查询"
- "设计一个智能家居中控 App，能控制灯光/空调/窗帘"

## 输出规范

生成完整的 PRD，必须包含以下 8 个部分：

### 1. 项目概述
- 项目名称
- 一句话定位
- 核心价值主张
- 目标用户画像（至少 3 类角色）

### 2. 业务背景
- 现状与痛点
- 解决的核心问题
- 竞品对比（如有）

### 3. 功能需求（MoSCoW 分级）
```
Must Have (必须有):
- F1: ...
- F2: ...

Should Have (应该有):
- S1: ...

Could Have (可以有):
- C1: ...

Won't Have (本次不做):
- W1: ...
```

### 4. 非功能需求
- 性能：响应时间 / 并发量
- 安全：认证 / 授权 / 数据加密
- 可用性：SLA
- 可扩展性：未来扩展点

### 5. 用户故事初稿（至少 5 条）
```
作为 <角色>，我希望 <功能>，以便 <价值>
```

### 6. 领域概念识别
列出核心业务实体及其关系（为后续 Class Diagram 做准备）：
```
| 实体 | 属性（关键） | 关联 |
|------|-------------|------|
| User | id, name, role | 1:N → Order |
| ... | ... | ... |
```

### 7. 验收标准
每个 Must Have 功能至少 1 条 Given-When-Then 验收标准

### 8. 里程碑计划
```
Phase 1 (MVP): 核心功能 / 预计工期
Phase 2: 增强功能 / 预计工期
Phase 3: 优化 & 扩展
```

## OO 原则检查清单

生成 PRD 后必须 self-check：
- [ ] 是否清晰区分了 Actor（用户角色）？
- [ ] 领域实体是否有明确的属性和关联？
- [ ] 功能需求是否可以用 Use Case 进一步展开？
- [ ] 是否识别了系统的边界（什么在系统内/外）？

## 使用示例

用户输入：
> 做一个在线考试系统

输出：按上述 8 部分生成完整 PRD，自动填充合理的业务假设（如角色=学生/教师/管理员），标注假设项供用户确认。

## 注意事项

- 对用户未明确的部分，基于行业经验补全并标注 **[假设]**
- 功能分级严格使用 MoSCoW，MVP 只含 Must Have
- 领域概念识别是衔接 C02-C04 的关键桥梁，务必认真对待
